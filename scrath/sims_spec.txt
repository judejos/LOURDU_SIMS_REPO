1. System Overview 
SIMS (Student Intern Management System) is a full-stack, AI-augmented web application that acts 
as a lightweight ERP for managing the complete intern lifecycle — from entity setup and onboarding 
through daily operations, task execution, AI-powered coaching, performance evaluation, and exit. A 
standalone AI layer runs alongside every core module to automate insights, assist interns, and 
reduce manual effort for managers and mentors. 
 
1.1 Core Objectives 
• 
Manage the full intern lifecycle across multiple organizations from a single platform 
• 
Enforce granular role-based access — every user sees and does only what is permitted 
• 
Automate monitoring: attendance tracking, task deadlines, escalations, and performance metrics 
• 
Provide AI-powered features: performance analysis, resume tools, mock interviews, chatbot 
assistant, and smart recommendations 
• 
Generate certificates, reports, and analytics without manual effort

2.1 Permission Framework 
Every action is gated by the permission engine. Four top-level dashboard access flags are checked 
at route level: hasInternAccess, hasAssetAccess, hasAttendanceAccess, hasPayrollAccess. Within 
each dashboard, fine-grained permissions control individual actions (create, edit, delete, approve, 
export). The permission check is GET /Sims/user-permissions/ — called on every protected route 
mount. Data visibility is enforced server-side by scoping all queries to the requesting user's entity, 
role, and assigned interns/domains.

3.2 Backend Architecture 
The backend is a Django REST Framework application. All API endpoints are prefixed with /Sims/ 
for the main SIMS app and /api/ for the standalone Forms app. Authentication uses DRF Token 
auth. The backend handles all business logic, permission enforcement, data aggregation, certificate 
generation, and file storage. 
 
3.3 AI Architecture 
The AI Layer is a separate Django app (or microservice) that integrates with the Anthropic Claude 
API (claude-sonnet-4-5 model). AI endpoints are prefixed with /ai/ and are secured by the same 
token auth as the rest of the system. Every AI call includes the user's context (role, domain, metrics, 
recent history) as the system prompt so the model gives personalized, data-grounded responses.

3.4 API Layer 
The frontend communicates exclusively via a centralized Axios instance (src/services/api.js) which: 
• 
Attaches the auth token to every outgoing request header 
• 
Intercepts 401 responses globally and redirects to /loginpage, clearing stored tokens 
• 
Has a 30-second timeout to fail fast on unresponsive endpoints 
• 
Is organized into named API modules: authAPI, usersAPI, tasksAPI, attendanceAPI, assetsAPI, 
feesAPI, documentsAPI, certificatesAPI, feedbackAPI, notificationsAPI, formsAPI, 
dashboardAPI, and more 
• 
AI endpoints use same token auth — no separate AI login required 
• 
AI responses are streamed where applicable for real-time UX (chatbot, mock interview, resume 
builder) 
 
3.5 State Management 
State is managed entirely within React components using useState and useEffect hooks. There is 
no global state library (no Redux, no Zustand). Shared state is passed via props or React Context. 
Each dashboard shell manages its own active view, data cache, and loading state locally.

5. Authentication & Session Management 
5.1 Login Flow 
6. User visits /loginpage and submits credentials to POST /Sims/login/ — returns Token. 
7. Frontend stores token in localStorage. Username and role also stored. 
8. GET /Sims/user-permissions/ is called → user routed to their permitted dashboard. 
 
5.2 Route Protection 
All routes are wrapped in ProtectedRoute (checks token presence) or ProtectedDashboardRoute 
(additionally calls /Sims/user-permissions/ and redirects unauthorized users). 401 responses auto-
clear localStorage and redirect to /loginpage. 
 
5.3 Password Reset Flow 
9. User visits /Recovery — enters email — POST /Sims/password-reset/request/ sends OTP. 
10. User enters OTP — POST /Sims/password-reset/verify/. 
11. User sets new password — POST /Sims/password-reset/update/. 
 
5.4 Logout 
Logout calls POST /Sims/logout/, then clears localStorage (token, username, role) and redirects to 
/loginpage. The global 401 Axios interceptor also auto-logouts the user if any API call returns 
unauthorized, ensuring stale sessions are cleared immediately. 
 
5.5 Security Hardening 
• 
MFA (TOTP/OTP) for Manager, Lead, and Super Admin roles 
• 
Session management: active session list, force logout of specific sessions 
• 
Password complexity enforcement (min length, mixed characters) and expiry reminders 
• 
Role delegation with time expiry — delegated permissions auto-expire; all delegate actions are 
attribution-logged 
• 
Immutable audit log (append-only) for all write operations

10. Learning / Training Phase 
A dedicated pre-project phase isolated from active project work. Learning tasks do not count 
towards project productivity KPIs but are tracked separately for intern progress. 
 
10.1 Purpose 
• 
Provide structured onboarding education before assigning project tasks 
• 
Allow Mentors to assign practice exercises and reading material without affecting KPIs 
• 
Give Leads and Managers visibility into how quickly interns progress through the learning 
curriculum 
 
10.2 Learning Task Flow 
43. Mentor assigns learning tasks (Task Type = Learning) to a new intern. 
44. Intern accesses dedicated Learning section in their dashboard — separate from project tasks. 
45. Intern works through materials and marks each learning task complete. 
46. Mentor monitors progress (% completion) on their Intern Management Dashboard. 
47. Mentor marks learning phase complete → intern transitions to Active project status. 
 
10.3 Learning Path Assignment 
Mentors can assign an ordered sequence of learning tasks as a learning path. Sequential 
dependencies are enforced — task N+1 unlocks only after task N is completed. Interns cannot skip 
ahead. 
 
10.4 AI-Enhanced Learning (see Section 15.3) 
AI features overlay onto the learning phase: skill gap analysis, personalized learning path 
recommendations, auto-generated practice quizzes, and AI-powered concept explanations 
triggered from within learning tasks.

15. AI Features — Complete Specification 
SIMS integrates a dedicated AI layer backed by the Anthropic Claude API (claude-sonnet-4-5 
model). All AI features communicate via a secured /ai/ endpoint namespace using the same token 
authentication as the rest of the system. The AI layer is context-aware — every request includes the 
intern's profile, domain, metrics, and relevant history so the model can give personalized, data-
grounded responses. 
 
AI Architecture Principles 
• 
Every AI call includes the user's context (role, domain, metrics, recent history) as system 
context 
• 
Responses are streamed where real-time output improves UX (chatbot, interview feedback, 
resume suggestions) 
• 
AI-generated content is always presented as a suggestion — the user can accept, edit, or 
discard 
• 
All AI interactions are logged for audit and to allow Mentors/Managers to review AI 
recommendations 
• 
AI features are role-scoped: interns can access coaching tools; managers access analytics 
and insights 
• 
Sensitive data (salary, personal IDs) is never sent to the AI layer — only performance and 
task data

21. Security & Audit 
21.1 Authentication Security 
• 
DRF Token authentication: DRF Token — a unique token per user, invalidated on logout 
• 
Automatic token cleanup: 401 response from any API call clears localStorage and redirects to 
login 
• 
All routes are protected — unauthenticated requests are redirected to /loginpage 
• 
Permission check on every protected dashboard mount via /Sims/user-permissions/ 
• 
MFA (TOTP/OTP) for Manager, Lead, and Super Admin 
• 
Session management: active session list and force logout 
• 
Password complexity enforcement (min length, mixed characters) and expiry reminders 
 
21.2 Data Visibility Enforcement 
• 
Server-side scope enforcement: all data-fetching endpoints filter by the requesting user's entity, 
role, and assigned interns/domains 
• 
Mentors cannot query interns not in their assignment list 
• 
Leads cannot access data from other domains 
• 
Managers see full entity data; Super Admin sees all entities 
• 
AI endpoints enforce the same scoping — intern data is never sent to the AI layer for users who 
don't have visibility rights 
 
21.3 Audit Logging 
• 
Append-only audit log: user, entity, action, timestamp, old value, new value 
• 
Accessible via GET /Sims/logs/ (Manager) and /Sims/logs/{empId}/ (per-intern) 
• 
AI interaction audit log: all AI requests logged with user, feature, timestamp, and prompt hash 
• 
Role/permission change versioning 
 
21.4 Soft Delete 
Major entities (Users, Interns, Tasks, Assets, Documents) support soft delete — records are 
flagged as deleted (is_deleted) but retained in the database. 
• 
Soft-deleted records do not appear in normal list views 
• 
Authorized roles can restore soft-deleted records from the Recovery page 
• 
Permanent purge is a separate authorized action available only to Manager/Super Admin 
• 
Deleted users list: GET /Sims/deleted-users/ 
• 
Archiving is available for completed/dropped interns and closed projects to keep active views 
clean

23. Complete End-to-End Execution Flow 
The complete lifecycle of a SIMS deployment from first install to intern exit, in chronological 
operational order. 
 
Phase 1 — System Setup (Super Admin) 
89. Super Admin logs in and creates an Entity (company/academy name). 
90. Creates a Manager account and assigns them to the entity. 
91. Activates required modules (Attendance, Tasks, Payments, AI Features). 
 
Phase 2 — Organization Configuration (Manager) 
92. Creates Branches, Departments, and Domains. 
93. Configures working hours, shifts, leave policy, payment cycles, and AI feature toggles in 
Settings. 
94. Creates Lead and Mentor accounts via /admin/register. 
95. Creates Teams and assigns Mentors as team leads. 
 
Phase 3 — Intern Onboarding 
96. Intern submits onboarding form at /InternOnboarding. 
97. AI Onboarding Readiness Check automatically evaluates submission completeness and resume 
domain fit. 
98. Manager reviews (with AI readiness score) and approves the account. 
99. Manager sends login credentials to intern. 
100. 
Intern logs in, uploads required documents. AI evaluates uploaded resume. 
101. 
Staff reviews and approves documents. 
102. 
Manager assigns intern to Domain and Team. Issues assets. 
103. 
Intern status → Active. 
 
Phase 4 — Learning Phase 
104. 
Mentor assigns learning tasks. AI generates a personalized learning path based on 
resume skill analysis. 
105. 
Intern works through learning tasks. AI generates practice quizzes and explains concepts 
on demand. 
106. 
Intern uses AI Mock Interview for technical interview preparation. 
107. 
Intern uses AI Resume Builder to create a professional resume from SIMS data. 
108. 
Mentor reviews learning progress. AI flags if intern is behind pace. 
109. 
Learning phase marked complete → intern transitions to Active project status.
